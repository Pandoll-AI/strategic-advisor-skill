# Strategic Advisor Sample v0.2.0

## RAG 메모리 vs grep 메모리: 논쟁의 현재 위치

작성일: 2026-06-29  
용도: `strategic-advisor` v0.2.0 설명형 리포트 샘플  
톤: 한국어 존대말  
해석 메모: 사용자의 “양파 진영”은 문맥상 “양쪽 진영”으로 해석했습니다.

## 1. 한 줄 판단

개인용 에이전트나 작은 팀의 작업 메모리는 `Markdown + rg/grep + 명시적 요약 파일`을 기본값으로 두는 편이 더 안정적입니다. 다만 문서가 커지고 표현이 다양해지면 RAG를 보조 검색층으로 붙이는 hybrid 구조가 더 현실적입니다.

제품형 장기 메모리에서는 RAG와 grep 중 하나를 고르는 방식으로는 부족합니다. 실제 쟁점은 “정보를 어떻게 검색할 것인가”가 아니라 “기억을 어떻게 쓰고, 고치고, 폐기하고, 근거와 함께 다시 불러올 것인가”입니다.

## 2. 판단 배경

LLM 에이전트의 메모리는 단순한 검색 문제가 아닙니다. 사용자의 선호, 프로젝트 규칙, 작업 이력, 문서 지식, 코드 구조, 과거 의사결정이 모두 섞여 있기 때문입니다. 이 정보들은 성격이 다르기 때문에 하나의 검색 방식만으로 안정적으로 다루기 어렵습니다.

RAG는 의미적으로 비슷한 자료를 찾는 데 유리합니다. 사용자가 정확한 단어를 기억하지 못해도 관련 문서를 불러올 수 있습니다. 반면 grep이나 `rg`는 정확한 이름, 경로, 명령어, 규칙, 코드 심볼을 찾는 데 강합니다. 검색 결과가 왜 나왔는지도 비교적 투명합니다.

따라서 이 논쟁은 “RAG가 더 똑똑한가, grep이 더 단순한가”로 보면 좁아집니다. 더 정확한 질문은 “어떤 종류의 기억에 어떤 회수 방식을 붙이고, 그 결과를 어떻게 검증할 것인가”입니다.

## 3. 실제 결정

이번 결정은 메모리 시스템의 기본 구조를 정하는 문제입니다.

첫째, 작은 프로젝트나 개인 에이전트에서 기본 메모리 저장소를 무엇으로 둘지 결정해야 합니다. 둘째, 의미 검색이 필요한 순간에 RAG를 어느 범위까지 허용할지 정해야 합니다. 셋째, 검색된 기억이 실제로 현재에도 맞는지 확인하는 절차를 설계해야 합니다.

이 결정을 잘못하면 두 가지 문제가 생깁니다. grep만 쓰는 구조는 표현이 조금만 달라도 중요한 자료를 놓칠 수 있습니다. 반대로 RAG만 쓰는 구조는 그럴듯하지만 오래되었거나 부정확한 chunk를 “기억”처럼 불러올 수 있습니다.

## 4. 핵심 주장 1: 작은 메모리는 평문과 grep이 기본값으로 더 낫습니다

### 주장

개인용 또는 작은 팀의 에이전트 메모리는 먼저 평문 파일과 `rg/grep` 검색으로 설계하는 편이 좋습니다.

### 근거

Anthropic Claude Code 문서는 `CLAUDE.md`, `MEMORY.md`, topic file 같은 평문 메모리 구조를 설명합니다. Claude Code Agent SDK도 `Glob`, `Grep`, 파일 읽기, Bash 같은 파일 기반 도구를 기본 작업 도구로 제공합니다.

이 구조는 에이전트가 기억을 불러올 때 사람이 직접 확인할 수 있는 파일과 경로를 남깁니다. 어떤 문장이 기억으로 쓰였는지 추적하기 쉽고, 오래된 내용은 사람이 바로 수정하거나 삭제할 수 있습니다.

### 해석

작은 메모리에서는 검색 정확도보다 통제 가능성이 더 중요합니다. 사용자의 선호나 프로젝트 규칙은 “비슷한 의미”보다 정확한 문구와 현재성에 민감합니다. 예를 들어 “이 프로젝트에서는 run.sh를 개발용으로만 쓴다” 같은 규칙은 의미적으로 비슷한 여러 chunk보다 원문 위치가 더 중요합니다.

grep 기반 메모리는 이 점에서 강합니다. 검색어가 맞으면 결과가 빠르고, 결과의 이유가 명확합니다. 또한 메모리 파일 자체가 사람이 읽을 수 있는 문서이기 때문에, 에이전트가 잘못된 기억을 만들었을 때 교정 비용이 낮습니다.

### 한계

grep은 표현이 다르면 놓칩니다. “장기 기억”, “persistent memory”, “agent memory”처럼 서로 다른 표현이 같은 개념을 가리킬 때 정확 검색만으로는 회수율이 낮아질 수 있습니다.

또한 자료가 많아질수록 검색어를 잘 고르는 능력이 중요해집니다. 에이전트가 어떤 키워드로 찾아야 할지 모르면, 평문 메모리가 있어도 사용하지 못할 수 있습니다.

### 결정 영향

초기 설계는 평문 메모리로 시작하는 편이 좋습니다. 다만 파일 수가 늘어나거나 표현 변형이 많아지는 순간을 RAG 도입 기준으로 미리 잡아야 합니다.

이 샘플의 권장 기본값은 `MEMORY.md`와 `memory/topics/*.md`입니다. 검색은 `rg`, `Glob`, 경로 규칙으로 시작하고, 검색 결과에는 파일 경로와 근거 문장을 남기도록 설계하는 편이 추적 가능성을 높입니다.

## 5. 핵심 주장 2: RAG는 메모리 자체가 아니라 회수층으로 보는 편이 안전합니다

### 주장

RAG를 “에이전트 메모리”라고 부르기보다, 큰 자료에서 관련 후보를 찾는 retrieval layer로 보는 편이 더 정확합니다.

### 근거

Letta의 “RAG Is Not Agent Memory”는 RAG와 agent memory를 구분합니다. 이 글은 RAG가 외부 지식 검색에는 유용하지만, 에이전트가 지속적으로 상태를 업데이트하고 기억을 관리하는 문제와는 다르다고 설명합니다.

MemGPT 논문도 제한된 context window 밖에 여러 memory tier를 두고, LLM이 무엇을 context로 올릴지 관리하는 방향을 제안했습니다. 이 접근은 단순 vector search보다 “기억 관리” 자체에 초점을 둡니다.

### 해석

RAG는 관련 문서를 찾아오는 데 강하지만, 기억의 생명주기를 책임지지는 않습니다. 기억에는 생성, 갱신, 충돌 해결, 폐기, 권한, 시간성, 출처가 포함됩니다. 단순 vector DB에 chunk를 넣고 top-k로 가져오는 방식은 이 생명주기를 충분히 다루지 못합니다.

특히 사용자 선호나 프로젝트 규칙은 시간이 지나면서 바뀝니다. “예전에는 A를 선호했지만 지금은 B로 바뀌었다”는 정보는 의미적으로 가까운 chunk 하나를 찾는 문제가 아닙니다. 어느 정보가 최신인지, 어떤 결정이 취소되었는지, 어떤 출처를 더 신뢰해야 하는지를 함께 판단해야 합니다.

### 한계

RAG를 retrieval layer로 낮춰 부르면, 실제 구현에서 별도 memory manager가 필요합니다. 이는 설계 비용을 올립니다. 작은 프로젝트에서는 과한 구조가 될 수 있습니다.

또한 RAG가 언제나 부정확하다는 뜻은 아닙니다. 문서가 많고 표현이 다양하며 사용자가 정확한 키워드를 모르는 환경에서는 RAG가 grep보다 훨씬 좋은 후보를 가져올 수 있습니다.

### 결정 영향

제품형 장기 메모리를 만들 때는 RAG를 핵심 저장소로 두지 않는 편이 좋습니다. RAG는 후보 검색층으로 두고, 실제 기억은 구조화된 기록과 검증 가능한 출처, 수정 이력, 폐기 정책을 갖춘 별도 계층으로 설계해야 합니다.

작은 프로젝트에서는 이 구조를 축소해도 됩니다. 예를 들어 평문 메모리를 authoritative source로 두고, 필요할 때만 embedding index를 만들어 fuzzy retrieval에 활용하는 방식이 현실적입니다.

## 6. 핵심 주장 3: 현재 논쟁은 hybrid memory 쪽으로 이동하고 있습니다

### 주장

현재 논쟁의 중심은 “RAG냐 grep이냐”가 아니라, 정확 검색과 의미 검색과 상태 관리를 어떻게 결합할지로 이동하고 있습니다.

### 근거

Anthropic의 Contextual Retrieval 글은 embedding만 쓰기보다 BM25와 reranking을 결합하는 접근을 설명합니다. 이는 의미 검색과 어휘 기반 검색을 함께 쓰는 방향입니다.

arXiv의 grep retrieval 평가 논문은 특정 실험 조건에서 grep이 vector retrieval보다 높은 정확도를 보인 사례를 보고합니다. 동시에 harness와 tool-calling 방식이 결과에 큰 영향을 준다고 말합니다. 이 논문은 grep이 항상 이긴다는 결론보다, LLM이 도구를 어떻게 쓰는지가 검색 품질에 중요하다는 점을 보여주는 자료로 읽는 편이 적절합니다.

GrepRAG도 그래프 구조와 textual retrieval을 결합해 성능을 높이려는 방향입니다. Oracle의 “From RAG to Memory Systems”와 2026년의 “Beyond RAG for Agent Memory”도 단순 RAG를 넘어 stateful memory system으로 이동하는 문제의식을 공유합니다.

### 해석

hybrid 방향은 실무적으로 자연스럽습니다. 정확한 이름과 경로가 있는 정보는 grep이 잘 찾습니다. 의미는 비슷하지만 표현이 다른 정보는 vector search가 잘 찾습니다. 검색 결과가 많을 때는 reranking이 필요합니다. 그리고 장기 기억은 검색 이전에 저장, 수정, 삭제 정책이 필요합니다.

이 흐름은 검색 엔진의 오래된 교훈과도 닮아 있습니다. 좋은 검색은 하나의 알고리즘이 아니라, 색인, 질의 확장, ranking, filtering, feedback이 결합된 시스템입니다. 에이전트 메모리도 같은 방향으로 가고 있습니다.

### 한계

hybrid는 복잡합니다. 작은 프로젝트에 처음부터 vector DB, graph memory, reranker, summarizer, decay policy를 모두 넣으면 유지보수 비용이 검색 이득보다 커질 수 있습니다.

또한 hybrid라고 해서 자동으로 정확해지는 것은 아닙니다. 검색 후보가 많아질수록 에이전트가 어떤 근거를 채택했는지 설명해야 합니다.

### 결정 영향

초기 구현은 작게 시작하고, 실패 유형이 확인될 때마다 보조층을 붙이는 편이 좋습니다. grep이 놓치는 질문이 반복되면 RAG를 붙입니다. RAG가 부정확한 chunk를 가져오면 reranking과 출처 검증을 붙입니다. 오래된 기억이 문제를 만들면 memory lifecycle을 설계합니다.

이 샘플에서 가장 방어적인 잠정 결론은 “평문 메모리 우선, 필요 시 semantic retrieval 보강, 최종 답변은 출처 기반으로 검증”입니다.

## 7. 반대 관점과 약점

RAG 우선 진영의 강한 반론은 분명합니다. 사람은 항상 정확한 키워드로 기억하지 않습니다. 특히 고객 대화, 회의록, 리서치 노트, 정책 문서처럼 표현이 넓게 퍼진 자료에서는 grep만으로는 회수율이 낮아질 수 있습니다. 이 반론은 맞습니다.

하지만 이 주장이 곧 “모든 메모리를 RAG에 넣자”로 이어지지는 않습니다. 의미 검색은 후보를 넓게 찾는 데 유리하지만, 최신성, 우선순위, 취소된 결정, 사용자의 현재 선호를 자동으로 해결하지 않습니다. 따라서 RAG는 후보 검색에는 강하지만 authoritative memory로 두기에는 위험합니다.

grep 우선 진영의 약점도 분명합니다. 너무 단순한 파일 검색만 믿으면, 표현이 다른 중요한 정보를 놓칩니다. 또한 큰 문서 묶음에서는 검색어를 바꿔 가며 여러 번 찾아야 하므로 토큰과 시간이 늘어납니다. 이 때문에 grep 우선 구조도 semantic retrieval을 완전히 배제하면 안 됩니다.

결론적으로 양쪽의 강점은 서로 배타적이지 않습니다. grep은 검증 가능한 정확 검색에, RAG는 넓은 후보 회수에, memory manager는 기억의 생명주기 관리에 각각 쓰는 편이 더 안전합니다.

## 8. 의사결정 게이트

Scope: Pass  
질문을 “어떤 검색 방식이 더 좋은가”가 아니라 “어떤 기억 유형에 어떤 회수 방식을 붙일 것인가”로 바꾸면 판단 범위가 명확해집니다.

Evidence: Partial  
공식 문서, 공급사 글, 논문을 확인했지만 특정 제품에서의 우열은 워크로드별 벤치마크가 있어야 합니다. 따라서 일반 권고는 가능하지만, 특정 시스템의 최종 설계는 실험으로 확정해야 합니다.

Freshness: Pass  
2026년 6월 기준 공개 문서와 최근 논문을 기준으로 정리했습니다.

Risk: Partial  
RAG와 grep 모두 잘못된 검색 결과를 확신 있게 사용하는 위험이 있습니다. 따라서 검색 결과에는 출처 위치와 채택 이유가 함께 남아야 합니다.

Feasibility: Pass  
작은 실험은 1주 안에 가능합니다. Markdown + rg 기준선과 간단한 embedding retrieval prototype을 같은 질문 세트로 비교하면 충분히 방향을 잡을 수 있습니다.

## 9. 추천 아키텍처

기본 구조는 다음처럼 잡는 편이 좋습니다.

```text
curated markdown memory
-> grep / rg / structural search
-> semantic retrieval for fuzzy recall
-> reranking
-> source-grounded answer
-> review / revise / forget loop
```

첫 계층은 사람이 읽고 수정할 수 있는 평문 메모리입니다. 여기에는 사용자 선호, 프로젝트 규칙, 결정 로그, 반복되는 작업 지침을 넣습니다. 이 계층이 authoritative source가 되어야 합니다.

두 번째 계층은 정확 검색입니다. 파일명, 경로, 명령어, 정책 이름, 심볼, 날짜처럼 정확한 단서가 있는 정보는 `rg`, `Glob`, 구조 검색으로 찾습니다. 이 계층은 빠르고 설명 가능해야 합니다.

세 번째 계층은 의미 검색입니다. 사용자가 정확한 표현을 모르거나, 문서가 많고 표현이 다양한 경우에만 RAG를 붙입니다. 이때 RAG 결과는 “정답”이 아니라 “후보”로 취급해야 합니다.

마지막 계층은 검토와 폐기입니다. 오래된 기억, 서로 충돌하는 기억, 출처 없는 요약은 장기 기억으로 승격하지 않는 편이 좋습니다. 에이전트가 자동으로 기록하더라도 사람이 검토할 수 있는 형태를 유지해야 합니다.

## 10. 7일 검증 계획

Day 1에는 실제 메모리 질문 30개를 만듭니다. 질문은 정확 키워드형, 의미 검색형, 시간/상태 추적형으로 나눕니다. 이 분류가 있어야 grep과 RAG가 어디서 실패하는지 볼 수 있습니다.

Day 2와 Day 3에는 `Markdown + rg` 기준선을 만듭니다. 같은 질문을 RAG prototype에도 넣습니다. 두 방식 모두 답변뿐 아니라 근거 위치를 남기게 합니다.

Day 4에는 정답률, 근거 위치 표시율, 토큰 사용량, latency를 비교합니다. 여기서 중요한 지표는 “그럴듯한 답”이 아니라 “근거를 다시 확인할 수 있는 답”입니다.

Day 5에는 실패 사례를 분류합니다. grep이 실패한 경우는 검색어 불일치인지, 파일 구조 문제인지, 기억 자체가 없는지 나눕니다. RAG가 실패한 경우는 chunk 오염, 오래된 정보, 중복 검색, 근거 누락을 따로 봅니다.

Day 6에는 hybrid 규칙을 만듭니다. 예를 들어 정확한 파일, 명령, 정책은 grep을 먼저 쓰고, 추상 질문이나 표현이 다양한 자료는 RAG를 먼저 쓰는 식입니다.

Day 7에는 같은 30문항을 다시 돌려 개선폭을 확인합니다. 개선폭이 작으면 검색 방식보다 메모리 저장 형식을 먼저 고쳐야 합니다.

## 11. 중단 기준

RAG가 근거 없는 chunk를 반복적으로 불러오면 top-k 기반 메모리 설계를 중단해야 합니다. 이 경우 reranking이나 source validation 없이 RAG를 장기 기억처럼 쓰면 위험합니다.

grep 방식이 질문의 30% 이상에서 검색어 설계 실패를 보이면 semantic retrieval을 붙이는 편이 좋습니다. 이는 grep이 나쁘다는 뜻이 아니라, 사용자의 질문 표현과 메모리 문서 표현이 충분히 맞지 않는다는 신호입니다.

양쪽 모두 출처 위치를 남기지 못하면 검색 방식보다 메모리 저장 형식을 다시 설계해야 합니다. 에이전트 메모리에서 가장 위험한 상태는 “답은 나오지만 왜 나왔는지 설명할 수 없는 상태”입니다.

## 12. 최종 권고

지금 개인 에이전트나 작은 팀의 메모리를 만든다면 grep 기반 평문 메모리부터 시작하시는 편이 좋습니다. 이 방식은 단순하고, 디버깅하기 쉽고, 사용자가 직접 수정할 수 있습니다.

제품형 장기 메모리를 만든다면 RAG를 메모리 전체로 설계하지 않는 편이 좋습니다. RAG는 retrieval layer로 두고, 실제 메모리는 쓰기, 수정, 폐기, 출처, 시간, 권한을 관리하는 별도 계층으로 설계하셔야 합니다.

따라서 현재의 실용적 결론은 hybrid입니다. 시작은 평문과 grep으로 작게 하고, 실패 패턴이 확인될 때마다 RAG, reranking, memory lifecycle을 추가하는 방식이 가장 방어적이면서도 확장 가능합니다.
